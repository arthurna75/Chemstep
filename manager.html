Chemstep 관리 문서

프로젝트 오너가 반복적으로 챙겨야 하는 항목 정리. 코드 세부사항이 아니라 "무엇을 설정하고, 무엇을 점검하고, 무엇이 위험한지"에 초점을 둡니다. 원본 마크다운은 manager.md.

Next.js (App Router) Supabase Vercel github.com/arthurna75/Chemstep · main

1 환경변수 관리

변수용도설정 위치
NEXT_PUBLIC_SUPABASE_URLSupabase 프로젝트 접속 주소Vercel → Project Settings → Environment Variables, 로컬은 .env.local
NEXT_PUBLIC_SUPABASE_ANON_KEYSupabase 익명 접근 키(공개 키, 진짜 시크릿 아님)위와 동일

이 두 개가 전부입니다. 서비스 롤 키나 외부(결제/이메일/AI 등) API 키는 코드 어디에도 없습니다. 두 값 다 없어도 앱이 죽지 않고 홈페이지에 "Supabase 연결이 필요합니다" 안내가 뜹니다. 값은 Supabase 대시보드 → Project Settings → API에서 확인합니다.

2 데이터베이스 관리

supabase/schema.sql 기준 — chapters → lessons → quizzes → quiz_options(콘텐츠, 전체 공개 읽기), user_progress / user_answers(본인 데이터만 접근), guest_codes(게스트 조회 코드). 모든 테이블에 RLS가 활성화되어 있습니다.

⚠ 마이그레이션 추적 없음 supabase/ 안에 마이그레이션 폴더가 없습니다. 스키마를 바꾸면 schema.sql을 수정한 뒤 Supabase SQL Editor에 수동으로 붙여넣어 실행해야 하고, 운영 DB에 무엇이 실제로 반영돼 있는지 추적할 방법이 없습니다.
⚠ schema.sql이 실제 DB 상태와 불일치 schema.sql의 시드 INSERT는 5개 단원만 넣지만, 실제로는 seed_chapter1.sql ~ seed_chapter8_*.sql까지 8개 단원 콘텐츠가 별도 실행되어 있습니다. schema.sql만으로 새 환경을 구축하면 3개 단원이 빠집니다.

시드 파일 현황: seed_chapter1.sql은 구버전, seed_chapter1_v2.sql이 최종본(1단원은 v2만 실행). 나머지는 챕터당 파일 하나씩.

3 콘텐츠 발행 워크플로우

새 단원/레슨 추가나 기존 콘텐츠 수정은 /new-chapter 스킬(.claude/commands/new-chapter.md)이 유일한 공식 절차입니다.

  1. Supabase에서 chapter_id 확인
  2. 레슨 5개 설계 (기초 → 심화)
  3. supabase/seed_<슬러그>.sql 작성
  4. 레슨별 SVG 일러스트 작성
  5. illustrations/index.ts 매핑 추가
  6. 비교 데이터는 ChemTable 기반 Visuals로
  7. npm run build 통과 확인
  8. git commit && git push (→ Vercel 자동 배포)
  9. Supabase SQL Editor에 seed SQL을 직접 붙여넣어 실행 — 빠뜨리면 코드는 배포됐는데 콘텐츠는 안 보임

4 인증 & 게스트 코드

이메일/비밀번호 가입만 지원합니다(OAuth 없음). 방문자는 로그인 없이도 익명 계정으로 자동 시작해 학습 진행상황이 쌓입니다. /progress에서 게스트 코드를 발급받아, 로그인 없이 코드만으로 진행상황을 조회할 수 있습니다(get_progress_by_code RPC, 읽기 전용). 나중에 회원가입하면 기존 익명 계정이 실제 계정으로 업그레이드되어 진행상황이 유지됩니다.

⚠ 게스트 코드는 비밀번호처럼 취급 코드를 아는 사람은 누구나 그 사용자의 진행상황을 볼 수 있습니다.

5 배포 파이프라인

CI 없음. main에 push하는 순간 Vercel이 자동으로 빌드/배포합니다(테스트·린트 게이트 없음). 커스텀 도메인/robots.txt/sitemap 설정은 없으며 별도로 관리할 것도 없습니다.

  • push 전에 로컬에서 npm run build(타입 에러 확인) 실행
  • push 전에 npm run lint 실행
ℹ AI 작업 정책 (AGENTS.md) Claude가 코드/문서를 변경하면 별도 요청 없이도 관련 문서 갱신 → npm run build 검증 → 커밋 → main push(배포)까지 기본으로 수행합니다. 특정 변경만 커밋해두고 배포는 미루고 싶다면 그때그때 명시적으로 범위를 좁혀 요청해야 합니다.

7 알려진 리스크 / 기술부채 체크리스트

  • DB 마이그레이션 추적 시스템 없음 (섹션 2)
  • schema.sql이 실제 운영 DB(8단원)와 불일치 (섹션 2)
  • CI/테스트/린트 자동 게이트 없음 — push 즉시 배포 (섹션 5)
  • 콘텐츠 발행에 관리자 UI 없음 — SQL Editor 수동 실행이 유일한 경로 (섹션 3)
  • README.md가 아직 create-next-app 기본 문구 그대로 — 신규 협업자는 이 문서를 대신 참고

8 정기 점검 체크리스트

  • 새 단원/레슨 배포 후 실제 사이트에서 콘텐츠가 보이는지 확인 (SQL Editor 실행 누락 여부)
  • Supabase 대시보드에서 사용량/쿼터(DB 용량, Auth 사용자 수 등) 확인
  • Vercel 대시보드에서 최근 배포 상태(빌드 실패 여부) 확인
  • guest_codes 테이블에 비정상적으로 많은 코드가 생성되고 있지 않은지 확인
  • schema.sql과 운영 DB가 계속 동기화되고 있는지 재확인
